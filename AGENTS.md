# AGENTS.md

このファイルは、このソリューションで作業する AI コーディングエージェント向けの指示です。

## 最重要ルール

- ユーザーが明示的にファイル変更を許可した場合にのみ、コード、設定、ドキュメント、テスト、生成物を編集してください。
- 「調査して」「確認して」「説明して」「レビューして」などの依頼は、原則として読み取り専用で対応してください。
- 変更が必要だと判断した場合でも、先に変更方針と対象ファイルを説明し、ユーザーの許可を得てから編集してください。
- 例外は、ユーザーが明示的に「作成して」「修正して」「実装して」「追加して」「削除して」など、ファイル変更を伴う作業を依頼した場合です。
- 既存のユーザー変更を勝手に戻さないでください。不要なリフォーマットや無関係な差分を作らないでください。

## プロジェクト概要

- C# / ASP.NET Core Web API のソリューションです。
- ターゲットフレームワークは `net10.0` です。
- Nullable reference types は有効です。
- Implicit usings は無効です。必要な `using` は明示してください。
- NuGet パッケージのバージョンは `Directory.Packages.props` で中央管理されています。
- .NET 10 では `.slnx` が既定のソリューションファイル形式です。このリポジトリでは `WatchStore.slnx` をソリューションとして扱い、`.sln` 前提のコマンドや説明をしないでください。
- API プロジェクトは `Api/WatchStore.Api.csproj` です。

## 作業姿勢

- まず既存コード、命名、ディレクトリ構成、依存関係、`.editorconfig` を確認し、既存パターンに合わせてください。
- 小さく局所的な変更を優先してください。依頼範囲外のリファクタリングは避けてください。
- 仕様が曖昧な場合は、勝手に大きな設計判断をせず、前提を明示して確認してください。
- セキュリティ、認可、入力検証、例外処理、ログ、永続化の変更は影響範囲を特に慎重に確認してください。
- 公開 API のレスポンス形式、HTTP ステータスコード、ルーティング、バリデーションエラーの形を変える場合は、互換性への影響を説明してください。

## C# コーディング規約

- `.editorconfig` に従ってください。
- ファイルスコープ namespace を優先してください。
- nullable 警告を抑制するためだけの `!` は避け、型設計、ガード、バリデーションで解決してください。
- `var` は既存規約に従い、型が明らかな場合に使用してください。
- 非同期 I/O には `async` / `await` と `CancellationToken` を適切に使用してください。
- 公開メンバーや DTO の名前は、API の意味が読み取れる具体的な名前にしてください。
- マジックナンバーや文字列は、必要に応じて定数、設定、型で表現してください。

## ASP.NET Core Web API 指針

- ルーティング、DTO、バリデーション、サービス登録は既存の構成に合わせてください。
- HTTP メソッドとステータスコードは REST の意味に沿って選んでください。
- リクエスト DTO とレスポンス DTO を分け、ドメイン内部の型を不用意に公開しないでください。
- 入力検証は FluentValidation など、既存の導入済みライブラリを優先してください。
- 例外を通常の制御フローに使わないでください。想定される不正入力や未検出は明示的に扱ってください。
- `CancellationToken` をエンドポイントから下位層へ伝播してください。
- ログには機密情報、認証情報、個人情報を出力しないでください。
- OpenAPI に出る型やエンドポイント名は、利用者が理解しやすい表現にしてください。

## 依存関係

- 新しい NuGet パッケージを追加する前に、既存の標準ライブラリ、ASP.NET Core 標準機能、導入済みパッケージで解決できないか確認してください。
- パッケージを追加する場合は、`Directory.Packages.props` にバージョンを追加し、プロジェクトファイルにはバージョンを書かないでください。
- 不要なフレームワーク、ORM、マッピング、認証、ロギング基盤を新規導入しないでください。

## 検証

- 変更を行った場合は、可能な範囲で次を実行してください。
  - `dotnet build WatchStore.slnx`
  - `just fmt`
- テストプロジェクトがある場合、関連テストを実行してください。
- 実行できなかった検証がある場合は、理由と未検証リスクを報告してください。

## コマンドと編集

- ファイル検索には `rg` または `rg --files` を優先してください。
- 手動編集は、対象ファイルを十分に読んでから最小差分で行ってください。
- 生成ファイル、ビルド成果物、IDE メタデータを不用意に変更しないでください。
- 破壊的な操作、マイグレーション、データ削除、依存関係更新、広範囲なフォーマットは、必ず事前にユーザー確認を取ってください。

## 回答方針

- 変更した場合は、変更ファイル、要点、実行した検証を簡潔に報告してください。
- 変更していない場合は、調査結果と次に必要な判断を明確にしてください。
- 不確かな点は推測として扱い、事実と分けて説明してください。
